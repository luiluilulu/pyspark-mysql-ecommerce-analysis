# 电商用户行为分析项目报告

## 1. 项目背景

本项目基于公开电商用户行为数据集，构建了一条离线数据处理与分析链路。项目目标不是单纯画图，而是模拟数据开发中的典型流程：原始行为日志进入本地环境后，先通过 PySpark 完成清洗与字段加工，再以 Parquet 保存全量明细，随后使用 Spark SQL 对全量数据进行聚合，将结果写入 MySQL 的 `ads_*` 结果表，最后用 Python + Plotly 生成可视化图表。

项目同时保留 MySQL 样本明细表 `user_behavior`，用于 SQL 练习和调试；正式分析结论优先来自 Spark 全量聚合后的 ADS 表。

## 2. 数据说明

原始数据字段包括：

| 字段 | 含义 |
| --- | --- |
| `user_id` | 用户 ID |
| `item_id` | 商品 ID |
| `category_id` | 商品类目 ID |
| `behavior_type` | 行为类型 |
| `timestamp` | Unix 时间戳 |

行为类型口径：

| 行为类型 | 含义 |
| --- | --- |
| `pv` | 浏览 |
| `fav` | 收藏 |
| `cart` | 加购 |
| `buy` | 购买 |

清洗后新增字段：

| 字段 | 含义 |
| --- | --- |
| `behavior_time` | 行为发生时间 |
| `behavior_date` | 行为日期 |
| `behavior_hour` | 行为小时 |

当前分析周期为 `2017-11-25` 至 `2017-12-03`。

## 3. 数据处理链路

项目采用两条链路：

```text
原始 CSV
    -> PySpark 全量清洗
    -> Parquet 明细层
    -> Spark SQL 全量聚合
    -> MySQL ads_* 结果表
    -> Plotly 可视化
```

以及：

```text
Parquet 明细层
    -> MySQL user_behavior 10000 行样本
    -> SQL 练习和小样本验证
```

这样设计的原因是：MySQL 更适合承载样本查询和小规模结果表，不适合在本地环境中直接承载 1 亿级行为明细；Spark 更适合读取 Parquet 并执行全量聚合。

## 4. 指标体系与口径

当前 ADS 层结果表由 `ads/sql/*.sql` 自动生成，表名规则为：

```text
ads_ + SQL 文件名去掉 .sql
```

核心指标如下：

| 指标 | ADS 表 | 口径 |
| --- | --- | --- |
| 行为类型分布 | `ads_behavior_type_count` | 按 `behavior_type` 统计行为量 |
| 每日 PV/UV | `ads_daily_pv_uv` | `pv` 行为量与去重浏览用户数 |
| 每小时行为量 | `ads_hourly_behavior_count` | 按 `behavior_hour` 统计总行为量 |
| 转化漏斗 | `ads_conversion_rate` | 浏览 -> 收藏/加购 -> 购买 |
| 复购率 | `ads_repurchase_summary` | 购买次数大于等于 2 的购买用户占比 |
| 类目热度 | `ads_category_top_analysis` | 按类目统计行为量、购买量、购买用户数 |
| 商品 TOP10 | `ads_top10_item_id` | 按商品总行为量排序 |
| RF 用户分层 | `ads_rf_segment_summary` | 用最近购买间隔和购买次数做用户分层 |

数据集没有金额字段，因此本项目不硬造消费金额，而是使用 RF 分层替代完整 RFM 分层。

## 5. 核心分析结果

### 行为类型分布

全量聚合结果如下：

| 行为类型 | 行为量 |
| --- | ---: |
| 浏览 `pv` | 89,660,688 |
| 加购 `cart` | 5,530,446 |
| 收藏 `fav` | 2,888,258 |
| 购买 `buy` | 2,015,839 |

浏览行为占绝对多数，购买行为量相对较低，符合电商用户从浏览到交易逐步收窄的典型行为结构。

### 每日 PV/UV

分析周期内，每日 PV 和 UV 在 `2017-12-02` 与 `2017-12-03` 明显升高：

| 日期 | PV | UV |
| --- | ---: | ---: |
| 2017-12-01 | 9,718,959 | 718,184 |
| 2017-12-02 | 12,329,644 | 939,383 |
| 2017-12-03 | 12,237,300 | 936,431 |

这说明周期末存在更高的用户活跃度，可能与促销、周末或活动周期有关。由于数据集不包含营销活动字段，当前项目只做行为层面的观察，不直接推断具体活动原因。

### 每小时行为量

行为量最高的时段集中在晚间：

| 小时 | 行为量 |
| --- | ---: |
| 21 | 8,369,714 |
| 22 | 8,299,812 |
| 20 | 7,304,809 |

晚间 20 点至 22 点是用户活跃高峰，适合用于运营触达、推荐曝光或活动推送等场景。

### 转化漏斗

当前漏斗按用户是否到达阶段统计，后一阶段用户必须属于前一阶段用户集合：

| 转化步骤 | 起始用户数 | 转化用户数 | 转化率 |
| --- | ---: | ---: | ---: |
| 浏览 -> 收藏/加购 | 984,105 | 855,498 | 86.93% |
| 收藏/加购 -> 购买 | 855,498 | 600,278 | 70.17% |

该漏斗是用户级宽松漏斗，不要求严格时间顺序。后续可继续升级为同一用户、同一商品维度下的有序漏斗。

### 复购分析

购买用户与复购用户统计如下：

| 指标 | 数值 |
| --- | ---: |
| 购买用户数 | 672,404 |
| 复购用户数 | 443,858 |
| 复购率 | 66.01% |

当前口径中，一条 `buy` 行近似代表一次购买行为；同一用户购买次数大于等于 2 即定义为复购用户。

### 类目热度 TOP5

按类目总行为量排序，TOP5 如下：

| 类目 ID | 总行为量 | 浏览量 | 收藏/加购量 | 购买量 | 购买用户数 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 4756105 | 4,860,459 | 4,477,682 | 354,519 | 28,258 | 23,754 |
| 4145813 | 3,467,459 | 3,150,716 | 285,085 | 31,658 | 27,529 |
| 2355072 | 3,373,265 | 3,151,734 | 209,108 | 12,423 | 10,523 |
| 3607361 | 3,169,516 | 2,976,359 | 180,436 | 12,721 | 10,623 |
| 982926 | 3,066,598 | 2,798,730 | 243,043 | 24,825 | 21,190 |

类目热度可以帮助识别高流量类目和高购买类目。后续可以进一步计算类目购买转化率，而不仅仅看总行为量。

### 商品 TOP5

按商品总行为量排序，TOP5 如下：

| 商品 ID | 总行为量 |
| --- | ---: |
| 812879 | 32,618 |
| 3845720 | 27,308 |
| 138964 | 22,973 |
| 2331370 | 22,031 |
| 3031354 | 20,393 |

这些商品具有较高的用户关注度，后续可以结合购买行为进一步区分“高曝光商品”和“高转化商品”。

### RF 用户分层

由于数据集没有金额字段，本项目使用 RF 分层：

```text
R = 最近一次购买距离分析基准日的天数
F = 用户购买次数
```

当前分层阈值基于 R 值分布确定，前 50% 购买用户最近购买间隔在 2 天以内，因此使用 `recency_days <= 2` 作为近期购买阈值。

| 用户分层 | 用户数 |
| --- | ---: |
| 高价值用户 | 295,667 |
| 沉默复购用户 | 148,191 |
| 普通购买用户 | 138,286 |
| 最近购买用户 | 90,260 |

高价值用户同时满足近期购买和复购行为，是后续运营中最值得重点维护的人群。

## 6. 可视化说明

`viz/visualize_analysis.py` 从 MySQL `ads_*` 表读取结果，并生成 Plotly HTML 图表。当前主要图表包括：

- `behavior_type.html`：行为类型分布柱状图。
- `daily_pv_uv.html`：每日 PV/UV 双 y 轴折线图。
- `hourly_behavior_count.html`：每小时行为量柱状图。
- `top10_items.html`：热门商品 TOP10 横向条形图。

图表默认输出到：

```text
viz/output/
```

该目录不提交 Git，运行脚本后本地生成。

## 7. 工程实现亮点

- 使用 PySpark 处理原始 CSV，并将清洗后的全量明细保存为 Parquet。
- 使用 Spark SQL 在全量 Parquet 上进行聚合，避免将大规模明细直接写入 MySQL。
- 使用 MySQL `ads_*` 表保存小规模分析结果，支撑可视化查询。
- 将路径、配置、Spark 会话、MySQL 连接、ADS 构建逻辑抽取到 `utils/`，降低重复代码。
- `ads/build_ads.py` 自动扫描 `ads/sql/*.sql`，按文件名生成对应的 `ads_*` 表。
- 保留 MySQL 样本明细表，用于 SQL 学习和口径验证。

## 8. 项目不足与后续优化

- 当前转化漏斗是用户级宽松漏斗，未严格校验同一商品和时间顺序；后续可升级为有序漏斗。
- 数据集缺少订单金额字段，因此无法完成完整 RFM，只能做 RF 分层。
- 当前可视化仍是静态 HTML 输出，后续可升级为 Streamlit Dashboard。
- 当前流程还不是一键 pipeline，后续可增加 `pipeline.py` 串联清洗、ADS 构建和可视化。
- 本地 Windows Spark 环境会出现临时 jar 删除 warning，生产环境通常应部署在 Linux、YARN 或容器化环境中。
